每个开发者都经历过的崩溃瞬间

🔥 "线上炸了，赶紧修！"

周三下午三点，你正在 feature/new-dashboard 分支上写得正嗨，CSS 调了半天终于对齐了，十几个文件改了一半还没提交。

群里突然 @所有人："生产环境登录接口 500，紧急修复！"

你只能赶紧 git stash，切到 hotfix 分支。修完 bug，切回来一看 -- stash 里的改动和当前代码冲突了，CSS 也全乱了。更惨的是，node_modules 因为两个分支的 package.json 不同，又要重新装一遍依赖...

用了 Worktree Manager：直接点"+"新建一个 hotfix worktree，两秒钟搞定。原来的代码一行不动，两个分支各跑各的，修完 bug 归档 hotfix 就好。

⏳ "又要等 npm install..."

你同时在做登录模块重构和首页性能优化。两个需求在不同分支上，每天在它们之间切换三四次。

每次 git checkout 之后：npm install 跑两分钟，Webpack 构建缓存全部失效再等三分钟，TypeScript 类型检查重新跑一遍... 一天下来，光等编译就花了一个多小时。泡咖啡的次数都不够填这些空档。

用了 Worktree Manager：两个 worktree 同时存在，node_modules 自动从主仓库链接共享，切换 worktree 只是在列表里点一下，零等待。两个 VS Code 窗口并排开着，左边改登录，右边优化首页。

👀 "帮我看一下这个分支的实现？"

Code Review 的时候，同事说："我在 feature/payment 分支上有个奇怪的 bug，你能帮我看看吗？"

你当前分支还有一堆未提交的改动。为了看他的代码，要么 stash + checkout（回来还得解冲突），要么新 clone 一份仓库。等终于切好了，同事已经去开会了...

用了 Worktree Manager：新建一个 worktree 检出同事的分支，秒级完成。或者更酷的方式 -- 让同事开启"远程分享"，你直接在浏览器里打开他的工作区，连终端都能共用，一起实时排查问题。

💾 "磁盘快满了..."

团队有个大型 monorepo，光 node_modules 就 2GB。为了同时跑两个分支的开发服务器，你 clone 了两份。4GB 依赖 + 两份代码 + 两份构建产物，256GB 的 SSD 已经亮红灯了。

更痛苦的是，每次有依赖更新，两边都要跑一遍 npm install。

用了 Worktree Manager：所有 worktree 共享同一个 .git 仓库数据，node_modules 通过软链接从主仓库共享。10 个分支并行开发，磁盘只占一份依赖的空间。

🌐 "你能共享一下你的屏幕吗？"

远程办公中，新同事遇到了环境配置问题。你想帮忙排查，但腾讯会议的屏幕共享延迟高、画面糊，看不清终端输出。他说了半天 "报错了"，你问 "什么报错"，截图发过来一看只截了半屏...

又或者，你想给异地的产品经理演示一下新功能的开发进度。为了这五分钟的演示，先开会议、共享屏幕、调分辨率，忙活了十分钟。

用了 Worktree Manager：点一下"分享"，把链接和密码发给对方。他在浏览器里打开就能看到你的完整工作区 -- 所有分支状态、代码目录、终端输出，全都实时同步。如果需要，还能直接在共享终端里一起敲命令，比屏幕共享好用一百倍。通过 ngrok 外网穿透，不在同一个网络也没问题。

Git Worktree Manager 让这些问题迎刃而解

利用 Git 原生的 worktree 功能，在同一个仓库中同时检出多个分支。每个分支有独立的工作目录，共享 .git 数据，还能自动链接 node_modules 等大文件夹，节省磁盘空间。更有一键远程分享和浏览器内置终端，让团队协作不受空间限制。

三步上手

简单几步，开始多分支并行开发

1

创建工作区（Workspace）

启动应用后，点击左上角的"新建 Workspace"按钮，选择一个目录作为工作区根目录。应用会自动初始化目录结构并识别其中的 Git 项目。

  • 新建工作区：选择一个空目录，应用会自动创建 projects/ 和 worktrees/ 目录
  • 导入已有项目：选择包含 Git 项目的目录，应用会自动检测并导入
  • 也可以通过 GitHub 简写（如 owner/repo）、SSH 或 HTTPS 地址添加新项目
2

新建 Worktree（工作分支）

点击侧边栏的"+"按钮，在弹出的对话框中完成以下设置：

  • 输入分支名（如 feature/login），支持 feature/、hotfix/ 等命名规范
  • 选择需要包含的项目（可以多选）
  • 选择基于哪个分支创建（默认 main）
  • node_modules 等配置好的文件夹会自动从主仓库链接，无需重新安装依赖
3

开始开发

在列表中点击切换 worktree，用你喜欢的 IDE 一键打开。各分支的代码、依赖、终端完全隔离，互不干扰。

  • 点击 IDE 按钮，快速用 VS Code、Cursor、IDEA 等打开项目
  • 使用内置终端运行命令，无需切换窗口
  • 实时查看分支状态：提交数、未推送变更、是否已合并到测试分支
  • 完成开发后归档 worktree，保持工作区整洁

工作区目录结构

├── .worktree-manager.json        # 工作区配置
├── projects/                     # 主仓库（main 分支）
│   ├── frontend/
│   └── backend/
├── worktrees/                    # Worktree 目录（自动创建）
│   ├── feature-login/
│   │   ├── projects/
│   │   │   ├── frontend/         # 独立分支的工作目录
│   │   │   └── backend/
│   │   ├── .claude → ../../.claude   # 自动链接
│   │   └── CLAUDE.md → ../../CLAUDE.md
│   └── hotfix-bug/
│       └── ...
├── .claude/
└── CLAUDE.md
